### Tuning Dashboard & tuning_overrides System — How to Log Changes
**Source:** [[agents/claude-code]]
**Date:** 2026-04-01T14:30:00
**Type:** improvement

The trading system now has a centralized tuning tracking system that connects parameter changes to trade performance. ALL agents that adjust parameters MUST use this system.

#### What exists
- **tuning_overrides table** in `trading_forex.db` — 37 records covering every tuning change from 2026-03-24 to present
- **tuning_logger.py** at `Forex Trading Team/Source/tuning_logger.py` — one-liner to log any change
- **Tuning dashboard** in admin Performance panel — shows batched changes with before/after trade metrics
- **API endpoints**: `/api/trading/tuning/performance`, `/api/trading/tuning/history`, `/api/trading/tuning/snapshot`

#### How to log a tuning change (MANDATORY for all agents)
```python
from tuning_logger import log_tuning_change

log_tuning_change(
    param="gate.oscillator_direction_stoch_sell",  # dotted path: category.param_name
    value="45",                                     # new value (string)
    previous_value="35",                            # old value (string)
    reason="Trade #2795 AUD_USD slipped through at stoch=53.4. Widened from <35 to <45.",
    backtest_result={"dollars_saved": 16.50, "trades_saved": 1, "key_trades": ["#2795 AUD_USD"]},
    approved_by="Tim (post-mortem)"                 # or "system", "guardian", "scout", etc.
)
```

#### Parameter naming convention (dotted paths)
- `gate.*` — entry gates in trading_cycle.py (bb_width, oscillator, cooldown, story_score, etc.)
- `watch.*` — watch_manager.py parameters (trigger_threshold, ema_velocity, core_bonus criteria)
- `scout.*` — trade_scout.py features (stoch_cross, rsi_divergence, path_a, consolidation filter)
- `validator.*` — validator prompt/rules changes (phase_patterns, thesis_framing)
- `guardian.*` — position_guardian.py rules and thresholds (rule6, auto_attach_sl, layer weights)
- `snipe.*` — snipe execution params (sl_atr_mult, tp_atr_mult, fan_state caps)
- `manual_trade.*` — manual trade params (sl_atr_mult, retrace_trail_rate)
- `fix.*` — bug fixes that change behavior (timestamp parsing, key mismatches)

#### change_type field values
- `param_change` — threshold/multiplier value changed
- `new_gate` — new blocking gate added to entry pipeline
- `new_feature` — new detection or capability added
- `new_rule` — new guardian/exit rule
- `bug_fix` — fix that changes observable behavior
- `revert` — rolling back a previous change
- `removal` — removing a gate/feature

#### batch_label field
Group related changes with a descriptive batch label:
`"2026-03-29 Post-Audit Overhaul"`, `"2026-04-01 SL Widening"`, etc.

#### How performance is measured
The dashboard API automatically:
1. Pulls 48hrs of closed trades BEFORE the change timestamp
2. Pulls closed trades AFTER the change (up to next change on same param)
3. Filters by source type based on param name (snipe/gate/scout → snipe trades, manual → manual trades)
4. Computes: total trades, win rate, avg PnL, total PnL, SL-hit count
5. Shows batch-level aggregate AND per-change deltas

#### CRITICAL: What breaks if you DON'T log
- Dashboard shows no record of the change
- No before/after performance measurement
- Other agents can't see what changed or why
- Tim can't evaluate if tuning is helping or hurting
- Changes become invisible — the exact problem this system was built to solve

#### Future work needed
- Wire `log_tuning_change()` directly into code paths where params are modified at runtime (guardian adaptive thresholds, scout confidence adjustments)
- Add `tuning_override_id` column to `live_trades` so individual trades can be traced to the exact parameter state at entry time
- Learning loop sentry should auto-detect when a metric shift correlates with a tuning change timestamp
