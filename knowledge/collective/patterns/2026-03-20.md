### trevor-9b LoRA training — key lessons on MLX/Metal crashes
**Source:** agents/claude-code
**Date:** 2026-03-20T07:39:19
Key lessons from training 9B LoRA on M1 Max 64GB: (1) Metal ImpactingInteractivity crash is NOT OOM — it is macOS killing GPU command buffer for taking too long. Caused by long sequences (4448 tokens) even when truncated to 1024. Fix: filter training data, not reduce layers. (2) Resume-adapter-file does NOT restore optimizer state in MLX — starts gradients fresh on shifted weights, destabilizes Metal allocator. Always train from scratch. (3) Filtered dataset path: ~/jarvis/training_data/sessions/_lora_filtered_9b — 8428 examples, all under 4000 chars (~1000 tokens). The 606 removed examples were session dumps and vault doc retrievals, not coding CoT. (4) Stable config: 6 layers, lr 3e-5, batch 1, seq 1024, grad-checkpoint, filtered data. Peak mem 42.976GB stable. (5) Qwen3.5 35B-A3B is NEWER than Qwen3 32B — released Feb 2026 vs Sep 2025. Already on latest architecture.

### Phase 3 predictive system audit — input quality is everything
**Source:** agents/claude-code, agents/guardian
**Date:** 2026-03-20T20:00:00
**Tags:** audit-2026-03-20-phase3

Phase 3 audit revealed the root cause behind all validator failures: the TA agent was delivering raw indicator numbers instead of annotated chart pictures. The validator issued ZERO confirms across 34 cycles — every executed trade bypassed it via the watch→snipe path. Additionally, Gate 5a (setup validation from Phase 2) blocked 100% of snipe trades because watch context has empty setup_id, and the snipe list used old S-code format that didn't match V4 alert types. All 6 significant historical losses had broken R:R (~0.40).

The critical insight: the validator works perfectly when given Tim's manually annotated charts. This is an input problem, not an intelligence problem.

**Changes implemented:**
- TA agent prompt redesigned → 6-section annotated chart picture (EMA STATE, BB STATE, CANDLE TESTS, RSI, RETRACEMENT STATUS, CASCADE PHASE)
- Validator prompt overhauled → synthesis-before-checklist protocol, Phase 2.5 TRADE_NOW at score 7, no-chart fallback (max 9), precision snipe output (entry/invalidation/target)
- Snipe thresholds by setup type: retracement 0.75, breakout 0.80, reversal 0.85, continuation 0.90
- Chart delivery retry (2 attempts, 5s delay) + fallback mode
- Pre-pipeline confluence filter: blocks EARLY_WARNING into peaked/contracting/tangled fans with story_score < 40
- H4 alignment gate: flight_recorder logging + 4h stale data grace period
- Snipe list: 39 new V4-format entries replacing old S-code format
- Gate 5a: injects alert_type when setup_id is empty in watch context

**Architecture principles codified:**
- Scout = early warning | TA = objective chart annotator (not analyst) | Validator = fisherman (synthesizes story, writes precision snipes) | Guardian = profit capture + risk mgmt
- M15 primary, M1 micro-reads, H4 direction context only
- Quality of the snipe IS the quality of the trade
- No single indicator outweighs another — complete picture synthesis

**Infrastructure note:** CRO-mlx service (port 11500) down since ~15:51 UTC — needs investigation.

**Evidence:** 34 cycles, 0 confirms. 100% Gate 5a rejection on snipe path. 6 losses at R:R 0.40. Validator correct on Tim's charts. 39 snipe entries updated. 8 code changes across TA, validator, pipeline, and gate logic.

### Gate 1 structural contradiction — hard-blocking retracement setups before scoring
**Source:** agents/claude-code
**Date:** 2026-03-20T21:00:00
**Tags:** audit-2026-03-20-phase3-gate1

Gate 1 (fan state filter) was hard-blocking 61% of all cycles (22/36). Structural contradiction: Gate 2 rewards E100 retest with +15 pts (the fishing line setup), but Gate 1 hard-blocked peaked/contracting fan states — killing the fishing line setup before it could be scored. 27 high-quality retracement signals (story_score >= 40, named entry_type) were being destroyed.

**Fixes:**
- Retracement override path in Gate 1: is_retracement + fan_ordered + story_score >= 40 + named entry_type bypasses the fan-state block
- bars_since_cross widened from 20 bars (5h) to 30 bars (7.5h) — valid setups developing over 5-7h were being rejected as stale
- Guards kept: low-quality retracements (score < 40, entry_type=none) still blocked

**Key pattern:** When gates at different pipeline stages encode contradictory logic, the earlier gate silently vetoes what the later gate was designed to reward. Always audit gate interactions end-to-end, not in isolation.

**Evidence:** 22/36 cycles blocked at Gate 1 (61%). 27 high-quality retracement signals killed. Gate 2 E100 retest +15pts unreachable. Post-fix: override path + bars_since_cross 20→30.

### Trevor multi-tenant architecture — design pattern
**Source:** agents/claude-code
**Date:** 2026-03-20T00:00:00
**Full docs:** `.planning/trevor_multitenant_architecture.md` + `.planning/trevor_architecture_map.md`

**The pattern:** Schema work is done. Runtime threading is broken. User_id is available at every route handler but gets dropped at the first function call boundary.

**Shared vs Per-User:**
- SHARED: market candle data, macro intelligence snapshots, backtest data (8.5M rows), MLX model instances, agent registry
- PER-USER: OANDA credentials, snipe list, live trades, scout alerts, trading preferences, SSE stream

**Root cause:** `user_id=2` hardcoded in 5 places: `trading_api_routes.py` lines 1136, 1513, 3104, 3361 + `position_guardian.py`. Fix these 5 lines and multi-tenancy propagates automatically through `BrokerCredentials().get_connection(user_id=user_id)`.

**Critical gaps:**
1. user_id=2 hardcoded — trading runtime ⛔ BLOCKER
2. trade_scout.py global user_id=2 ⛔ BLOCKER
3. watch_suggestions has no user_id ⚠️
4. SSE broadcasts to all clients ⚠️
5. Boardroom conversations unsegregated ⚠️
6. Admin seed resets password on restart ⚠️
7. pair_last_close cooldown is shared across users ⚠️
8. PositionGuardian lacks user_id ⚠️

**What already works:** broker_credentials.py encryption, route auth, trading_preferences risk overrides, user_snipe_list schema — these are the patterns the rest of the system needs to follow.

**Architecture principle:** One server, one model pool, N user contexts. NOT N separate stacks. SQLite WAL mode + shared databases with user_id partitioning. Don't split into per-user files.

**Agent model implications (CRITICAL — see detail below):**
- execution agent: LLM **bypassed** in trading_cycle.py — direct Python `place_market_order()` wrapper call since "LLM adds zero value, costs ~60s, can silently fail"
- oanda_data agent: `_swarm_execute_tool` (legacy Python wrapper path) — may not go through LLM at all in current implementation; Haiku spec exists for MCP tool-calling scenarios
- trade_monitor agent: `_agent_task()` called ONCE (max_tokens=256) after trade fills; Sonnet 4.6 intentionally for distillation training data

**9B local model verdict:**
- oanda_data: The spec comment explicitly says "MLX/CRO does not handle tool calls reliably" — 9B cannot safely replace Haiku if LLM path is used
- execution: Already bypassed — moot
- trade_monitor: Simple enough call but loses distillation training value on Sonnet
