# 2026-03-25 — Learning Pipeline Resilience + Signal Accuracy Fix

## Key Patterns for All Agents

### 1. Two-Phase Commit for Multi-Stage Pipelines
When a pipeline has Stage A (reconcile → commit to DB) followed by Stage B (audit → learn → write vault), a crash between stages orphans the data. Solution: add a status column that transitions `NULL → 'pending' → 'complete'`. On next run, pick up anything stuck at 'pending', clean partial writes, re-run Stage B. This is now live in `outcome_reconciler.py` via `learning_status` column.

### 2. Idempotency for Counter-Based Metrics
Narrative vault writes (markdown append) are naturally safe to duplicate — extra context doesn't corrupt anything. But counter-based metrics (trades++, wins++, running averages) WILL corrupt on duplicate processing. Solution: track `processed_trade_ids` in the metric store. Skip if already seen. Cap the ID list to prevent unbounded growth.

### 3. Scout Context Survives Beyond Flight Recorder
For trades that go through the live scout→cycle→execution pipeline, flight_recorder has everything. But reconciled trades (matched post-hoc from OANDA) have no flight data. However, live_trades stores all scout context at entry time (fan_state, fan_direction, momentum_state, etc.). Any downstream consumer (auditor, learning integrator) should fall back to these columns when flight_data is empty.

### 4. Cowork Agent Now Vault-Wired
The Cowork (Claude Desktop) agent now reads from and writes to the knowledge vault, same as Claude Code and OpenClaw. Training data from Cowork sessions is captured in `collective/training-data/cowork-corrections-YYYY-MM-DD.jsonl` for 35B distillation. Agent learnings go to `agents/cowork/learnings.md`.
