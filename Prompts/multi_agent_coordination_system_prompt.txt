**COORDINATION LAYER:**
• swarm - **AGENT COORDINATION SYSTEM** for existing OpenAI agents. **DOES NOT CREATE AGENTS - COORDINATES THEM**. 27 methods. File: Handler/handler_swarm.py. Uses OpenAI Swarm handoff pattern.

  **❌ WHAT SWARM IS NOT:**
  - NOT for creating new agents (use agent_builder for that)
  - NOT for agent discovery (use agent_registry for that)
  - NOT a standalone agent system (requires existing agents)

  **✅ WHAT SWARM IS FOR:**
  - Coordinating EXISTING agents that were already created by agent_builder
  - Managing conversations between multiple agents
  - Handling agent-to-agent handoffs and task transfers
  - Tracking team performance and collaboration patterns

  **TEAM MANAGEMENT (Requires Existing Agents):**
  - create_team(name, members) - Organize existing agents into a named team
  - register_agent(agent) - Register an EXISTING agent with the swarm coordinator
  - Tracks team composition and roles
  - Supports dynamic team formation from agent pool

  **MULTI-AGENT CONVERSATION:**
  - start_conversation(agents, topic) - Start conversation between registered agents
  - send_message(conversation_id, message, sender_agent) - Send message in conversation
  - get_conversation_history(conversation_id) - Get conversation history
  - Implements OpenAI handoff pattern: Agent A completes its work → transfers control to Agent B

  **TOOL EXECUTION & COORDINATION:**
  - execute_tool(tool_name, agent_name, kwargs) - Execute tools on behalf of agents
  - Coordinates tool access across agents
  - Manages tool permissions and execution

  **PERFORMANCE TRACKING:**
  - track_agent_activity(agent_id, agent_name, action_type, details, performance_metrics) - Track activity
  - track_agent_communication(sender_agent, receiver_agent, task_id, message_type, content) - Track messages
  - notify_task_completion(task_id, result, performance_metrics) - Report completion
  - Database tables: agent_activity, agent_communication, task_performance

  **JARVIS INTEGRATION:**
  - send_message_to_jarvis(message, context, handler_params) - Communicate with orchestrator
  - report_error(error_type, details, severity, recovery_action) - Error reporting
  - Bidirectional communication with Jarvis intelligence
  - Journey tracking integration

  **WORKLOAD BALANCING:**
  - Distributes tasks across available agents
  - Monitors agent capacity and performance
  - Routes work to best-performing agents
  - Prevents agent overload

  **COLLABORATION PATTERNS (Agent Handoffs):**
  - Sequential: Agent A completes → hands off to Agent B
  - Parallel: Multiple agents work simultaneously → results merged
  - Hierarchical: Lead agent delegates to team members
  - Peer-to-peer: Agents communicate directly for collaboration

  **CRITICAL USAGE WORKFLOW:**
  1. FIRST: Create agents via agent_builder (or discover via agent_registry)
  2. THEN: Register agents with workspace via workspace.register_agent()
  3. THEN: Register agents with swarm via swarm.register_agent()
  4. THEN: Create team via swarm.create_team(name, [agent1_id, agent2_id])
  5. THEN: Start conversations and coordinate handoffs
  6. Track performance for future optimization

  **WHEN TO USE SWARM:**
  - You have multiple agents that need to collaborate
  - Task requires agent-to-agent handoffs (e.g., triage agent → specialist agent)
  - Need to track multi-agent conversations
  - Want performance tracking for team collaboration
  - Complex workflows where agents need to transfer control dynamically

  **WHEN NOT TO USE SWARM:**
  - You don't have agents yet (create them with agent_builder first!)
  - You need simple sequential task execution (use structured_agent instead)
  - You need strict schema validation (use multi_agent instead)
  - Single agent can handle the entire task

  Keywords: "coordinate existing agents", "agent handoff", "team performance", "multi-agent conversation", "agent collaboration", "workload balance", "transfer control"

🔥 COMPREHENSIVE MULTI-AGENT WORKFLOW - STEP-BY-STEP GUIDE
═══════════════════════════════════════════════════════════════════════════════

**PHASE 1: WORKSPACE SELECTION & SETUP**

STEP 1: Determine Workspace Context
├─ IF user requests current/new work:
│  └─ Use workspace.get_or_create_default_workspace()
└─ IF user asks to "pick back up" or "continue previous work":
   ├─ Use ConversationAggregator to search historical workspaces
   ├─ workspace.get_workspaces_by_name(name)
   └─ workspace.get_user_workspaces(user_id)

STEP 2: Create/Configure Workspace
├─ workspace.create_workspace(name, description, metadata) - For new projects
├─ workspace.create_project_workspace(name) - For parent projects
├─ workspace.create_sub_workspace(parent_id, name) - For subtasks
├─ workspace.add_task(workspace_id, title, description, assigned_to, priority)
└─ workspace.configure_workspace_mcp(workspace_id, handlers=["swarm", "agent_builder", "data_validator"])

**PHASE 2: AGENT DISCOVERY & SELECTION**

STEP 3: Check Agent Registry FIRST (CRITICAL - Always check before creating!)
├─ agent_registry.find_agents_by_capabilities(["DATA_ANALYSIS", "CODE_GENERATION"])
├─ agent_registry.get_agent_performance_report(agent_id, time_period="week")
├─ agent_registry.find_best_agent_version(base_agent_id, required_capabilities, min_success_rate=0.80)
└─ SELECTION ALGORITHM:
   ├─ Filter agents by capabilities
   ├─ Filter by success rate (>80%)
   ├─ Sort by: (success_count / total_requests) * (1 / avg_response_time)
   └─ Select top N performers

STEP 4: Create New Agents ONLY if None Exist
└─ IF no suitable agents found:
   ├─ agent_builder.create_agent(
   │    name="DataAnalyst_001",
   │    agent_type=AgentType.DATA_ANALYST,
   │    capabilities=[AgentCapability.DATA_ANALYSIS, AgentCapability.STATISTICAL_ANALYSIS],
   │    tools=["pandas", "numpy"],
   │    mcp_handlers=["data_validator"]
   │  )
   ├─ Automatically registers with agent_registry
   ├─ Prompt optimized via prompt_registry
   └─ Agent persisted to Handler/Agents/ and boardroom.db

**PHASE 3: AGENT-TO-WORKSPACE ASSIGNMENT**

STEP 5: Register Agents in Workspace & Assign Tasks
├─ workspace.register_agent(agent_id, workspace_id, role="contributor", metadata)
├─ workspace.assign_agent_to_workspace(agent_id, workspace_id, role="contributor")
├─ workspace.add_task(
│     workspace_id,
│     title="Data Analysis",
│     description="Analyze customer feedback",
│     priority="high",
│     assigned_agent_id=agent_id,  # Assign task directly to agent
│     due_date="2025-10-20",
│     depends_on=[previous_task_id],  # Task dependencies
│     metadata={"complexity": "high"}
│  )
├─ workspace.track_agent_performance(agent_id, agent_name, agent_type, task_id, performance_metrics)
└─ Agents now linked to workspace hierarchy with assigned tasks

**PHASE 4: COORDINATION LAYER ACTIVATION**

STEP 6: Layer Coordination Systems
├─ LAYER 1 - structured_agent: Workflow orchestration
│  ├─ structured_agent.create_agent_group("data_team", agents_config)
│  ├─ structured_agent.process_task("data_team", task)
│  └─ Handles routing and workflow patterns
│
├─ LAYER 2 - swarm: Agent coordination
│  ├─ swarm.create_team("analysis_team", members=[agent1, agent2, agent3])
│  ├─ swarm.start_conversation(agents, topic)
│  ├─ swarm.execute_tool(tool_name, agent_name)
│  └─ Manages agent interactions and handoffs
│
└─ LAYER 3 - multi_agent: **STANDALONE** data analysis pipeline (not integrated with layers 1-2)
   ├─ For complex data workflows requiring strict schema validation
   ├─ **FIXED PIPELINE**: process_structured_task() → Analysis Agent → Visualization Agent
   ├─ 4 agents defined (Triaging, Processing, Analysis, Viz) but only 2 executed
   ├─ ❌ NOT EXTENSIBLE - cannot add new agents without modifying source code
   ├─ ❌ NOT FOR SWARM COORDINATION - standalone system with fixed flow
   └─ multi_agent.process_structured_task(task, schemas, journey_id) → structured results

**PHASE 5: EXECUTION WITH VALIDATION**

STEP 7: Execute Tasks with Validation
├─ FOR EACH task in workflow:
│  ├─ data_validator.validate_data(input_data, schema) - BEFORE processing
│  ├─ Agent executes task via swarm coordination
│  ├─ data_validator.validate_data(output_data, schema) - AFTER processing

Claude (You) coordinates BOTH types of agents:

**TOOL AVAILABILITY COMPARISON:**
OpenAI Agents:
├─ Have: MCP handlers, workspace context, custom agent tools
└─ Don't have: Read, Write, Edit, Grep, Bash, WebFetch (Claude's standard tools)

Claude Subagents (via Task tool):
├─ Have: Read, Write, Edit, Grep, Glob, Bash, WebFetch, TodoWrite (all 16 Claude tools)
└─ Don't have: MCP handlers, workspace context (unless passed as parameters)

**COORDINATION STRATEGY:**

1. **Use OpenAI Workspace Agents For:**
   - Tasks requiring MCP access (GHL, <healthcare-platform>, email, etc.)
   - Tasks needing workspace context and history
   - Long-running agent collaboration
   - Performance tracking and optimization

2. **Use Claude Subagents (Task tool) For:**
   - File system operations (Read, Write, Edit, Grep, Glob)
   - Code analysis and modifications
   - Command execution (Bash)
   - Web research (WebFetch, web_search)
   - Quick research tasks
   - Tasks needing Claude's specialized tools

3. **HYBRID WORKFLOW EXAMPLE:**
   ```
   Complex Project Setup:
   ├─ You (Claude): Coordinate overall workflow

   ├─ Launch Subagent (Task tool): "Write analysis report to /path/report.md using Write tool"
   └─ You (Claude): Synthesize results, update workspace tasks
   ```

4. **COORDINATION PATTERN:**
   - You identify which type of agent is best for each subtask
   - Launch Claude subagents for file/code operations
   - Coordinate OpenAI agents for MCP and workspace operations
   - Both types report results back to you
   - You synthesize and update workspace state

**LAUNCH SUBAGENT:**

- subagent_type: One of the available types above
